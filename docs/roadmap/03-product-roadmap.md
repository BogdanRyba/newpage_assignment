# 03 · Продуктовий роадмап (Now / Next / Later)

> Питання: **що будувати і в якому порядку, щоб кожна фіча розблоковувала продаж, а не просто «покращувала продукт».**

## Принцип пріоритезації

Кожен пункт оцінюємо за двома осями:
- **Що розблоковує** — який сегмент / тип угоди стає можливим (прив'язка до [02](02-customers-and-use-cases.md) і [05](05-monetization.md)).
- **Складність** — груба оцінка (S / M / L), спираючись на наявні архітектурні шви.

Правило: **не будуємо фічу, яка не наближає платну угоду.** «Технічно круто» без покупця → в беклог.

Архітектурна підказка: завдяки hexagonal-lite, портам (`typing.Protocol`) і наскрізному `RepoContext`, багато enterprise-фіч (auth, tenancy, приватні LLM) лягають на існуючі шви без переписування. Це здешевлює «Next» і «Later».

---

## NOW (0–3 місяці) — «зробити pilot можливим»

Мета горизонту: довести цінність на реальному репо клієнта і зняти найгрубіші бар'єри першого платного pilot.

| Фіча | Що розблоковує | Складність |
|------|----------------|------------|
| **Інкрементальна реіндексація на git push** | Без цього на pilot індекс застаріває за дні → цінність падає. Обов'язкова умова «живого» інструмента. | M |
| **Багатомовний парсинг (Go, Java, C#)** | Зараз сильні Python/TS/JS. Enterprise-легасі — часто Java/C#. Без мови клієнта pilot неможливий. | M (tree-sitter граматики + чанкінг-стратегії за наявним патерном) |
| **Експорт відповіді з цитатами** (md/PDF) | Для due diligence і аудиту відповідь має лягати у звіт. Дешево, але відмикає сценарії B/C. | S |
| **Базові API-ключі + per-key квоти** | Мінімум контролю доступу/вартості, щоб дати інструмент команді на pilot без хаосу. | S–M |
| **Гардрейли «під навантаженням»** | Pilot на реальному великому репо тестує бюджети/таймаути/відмови — допиляти стійкість. | S |

**Definition of done горизонту:** можемо взяти legacy-репо клієнта, заіндексувати, тримати індекс свіжим і показати ROI за 1–2 тижні (див. pilot-стратегію в [06](06-go-to-market.md)).

---

## NEXT (3–9 місяців) — «зробити enterprise-угоду можливою»

Мета горизонту: зняти бар'єри, через які служба безпеки і procurement кажуть «ні». Деталі gap'ів — у [04](04-enterprise-readiness.md).

| Фіча | Що розблоковує | Складність |
|------|----------------|------------|
| **Auth + модель org/team/user** | Фундамент усього enterprise. Лягає поверх `RepoContext` як новий рівень ізоляції (org → repo). | L |
| **RBAC per-repo** | «Не кожен бачить кожен репозиторій» — базова вимога enterprise. | M (на тому ж seam, що й auth) |
| **Audit logs** | Хто що заіндексував / запитав / коли. Без цього немає SOC 2 і немає розмови з комплаєнсом. | M |
| **Admin-дашборд** | Керування репо, користувачами, використанням. Те, що показують на demo procurement'у. | M–L |
| **VSCode-плагін (тонкий клієнт над REST)** | Зустріти інженера там, де він живе → adoption усередині акаунта → розширення угоди. REST уже є, плагін тонкий. | M |
| **GitHub/GitLab інтеграція (Q&A у PR)** | Вбудувати цінність у наявний workflow рев'ю → щоденне використання → утримання. | M |

**Definition of done горизонту:** можемо пройти базовий security-questionnaire і підписати річний enterprise-контракт із кількома командами.

---

## LATER (9–18 місяців) — «масштабувати і поглиблювати рів»

Мета горизонту: великі контракти, регульовані галузі, операційна зрілість.

| Фіча | Що розблоковує | Складність |
|------|----------------|------------|
| **SSO/OIDC + SAML** | Тверда вимога великого enterprise. Без федеративної ідентичності у багато компаній не зайти. | L |
| **Приватні LLM (Llama/Ollama/Azure OpenAI) за портом generator** | «Дані й інференс не виходять із периметра взагалі». Найжорсткіша вимога фінтех/оборони. Порт `Generator` уже абстрагований. | M–L |
| **Data residency / multi-region** | Регуляторні вимоги до місця зберігання даних (GDPR тощо). | L |
| **Outbox / reconciliation (Postgres ↔ Qdrant)** | Закрити відомий gap консистентності при збої ingest. Операційна надійність для SLA. | M |
| **Cost attribution per-org/per-repo** | Біллінг, прозорість витрат, основа для usage-tiers. | M |
| **Поглиблення графа коду** (потоки даних, зони впливу змін) | Підсилює сценарії модернізації й безпеки — наш диференціатор довіри. | L |

---

## Що навмисно НЕ робимо (анти-роадмап)

Щоб не розпорошити ресурс:
- **Не змагаємось у autocomplete/генерації коду.** Це поле Copilot/Cursor; там ми програємо. Наш фокус — розуміння й довіра.
- **Не будуємо власну LLM/ембедери.** Використовуємо найкращі доступні моделі за портами; цінність — у пайплайні довіри, не в моделі.
- **Не вкладаємось у публічний multi-tenant SaaS як основу.** Він суперечить self-hosted-стратегії [05](05-monetization.md); тримаємо лише як легкий wedge.

---

## Зведена картина

```
NOW (pilot можливий)          NEXT (enterprise-угода)        LATER (масштаб + рів)
─────────────────────         ──────────────────────         ─────────────────────
Інкрем. реіндексація          Auth + org/team модель         SSO / SAML
Мови: Go/Java/C#              RBAC per-repo                  Приватні LLM (on-prem)
Експорт із цитатами           Audit logs                     Data residency
API-ключі + квоти            Admin-дашборд                  Outbox/reconciliation
Стійкі гардрейли             VSCode-плагін                  Cost attribution
                              GitHub/GitLab у PR             Глибший граф коду
```

Кожен горизонт відмикає наступний рівень монетизації: NOW → перший платний pilot; NEXT → річні enterprise-контракти; LATER → великі регульовані акаунти. Деталі цін — у [05](05-monetization.md).
