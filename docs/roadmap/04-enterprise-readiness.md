# 04 · Enterprise-готовність (gap-аналіз)

> Питання: **що в продукті є вже, чого немає, і яка саме відсутня річ блокує яку угоду.** Чесно, без прикрас — це внутрішній документ.

## Як читати цей gap-аналіз

Кожен gap прив'язаний не до «було б добре», а до конкретного **бар'єру продажу**:
- 🟥 **Блокує перший платний pilot** — без цього навіть демо на реальному коді не злетить.
- 🟧 **Блокує enterprise-контракт** — pilot можливий, але річний договір із великою компанією — ні.
- 🟦 **Блокує комплаєнс / великий регульований акаунт** — SOC 2, фінтех, оборона.

Хороша новина: архітектура побудована з правильними швами (hexagonal-lite, порти-`Protocol`, наскрізний `RepoContext`). Більшість gap'ів — це «дописати поверх seam», а не «переписати».

---

## Що вже є (наші сильні сторони)

Це фундамент, на якому будуємо, і це варто проговорювати на продажах:

| Можливість | Стан | Чому важить для enterprise |
|------------|------|----------------------------|
| Per-repo ізоляція через `RepoContext` | ✅ Працює | Готовий seam для org/tenant-ізоляції |
| Порти на волатильних IO-межах (embedder, generator, vector, graph) | ✅ Працює | Дозволяє підмінити на **приватні LLM** без переписування |
| Верифіковані цитати + детермінована перевірка | ✅ Працює | Це наш диференціатор довіри ([01](01-problem-and-value.md)) |
| Відмова при порожньому/недостатньому retrieval | ✅ Працює | Не вигадує → придатне для рішень, що коштують грошей |
| Ін'єкційний захист (код як дані, не інструкції) | ✅ Працює | Базова безпека проти ворожого вмісту в репо |
| Eval-харнес із порогами + cassettes | ✅ Працює | Доказова якість, детермінований CI — аргумент для технічного покупця |
| Граф коду (EXTENDS/IMPLEMENTS, виклики) | ✅ Працює | Основа сценаріїв модернізації/безпеки |
| Docker-compose розгортання | ✅ Працює | База для self-hosted; але не production-grade (див. нижче) |

## Чого немає: gap'и за пріоритетом бар'єру

### 🟥 Блокує перший платний pilot

| Gap | Наслідок для продажу | Спирається на seam |
|-----|----------------------|--------------------|
| **Базовий контроль доступу (API-ключі/квоти)** | Не можемо дати команді інструмент без хаосу й неконтрольованих витрат на LLM | Новий рівень в API + конфіг |
| **Інкрементальна реіндексація** | На pilot індекс застаріває → цінність «протухає» за дні | Ingest-пайплайн уже ідемпотентний (uuid5) — добудувати тригер на push |
| **Production-grade розгортання** (не dev-compose) | Зараз CORS `*`, reload-режим, ключі в `.env` — це dev. Безпека клієнта не пустить це в периметр | Потрібен hardened compose / Helm-чарт |
| **Мова кодової бази клієнта** (часто Java/C#) | Якщо не парсимо їхню мову — pilot неможливий | tree-sitter граматики за наявним патерном чанкінгу |

### 🟧 Блокує enterprise-контракт

| Gap | Наслідок для продажу | Спирається на seam |
|-----|----------------------|--------------------|
| **Auth + модель org/team/user** | Немає поняття «організація» і «користувач» → немає кому й як продавати seat'и | Новий рівень ізоляції **над** `RepoContext` |
| **RBAC per-repo** | «Не всі бачать усі репо» — базова вимога; без неї security каже «ні» | Той самий auth-seam |
| **Audit logs** | Procurement/security питають «хто що робив» — немає відповіді | Нова таблиця подій + middleware |
| **Admin-дашборд** | Немає що показати на demo для керування акаунтом | Поверх API + auth |
| **Secrets management** | Ключі в `.env` неприйнятні; потрібна інтеграція з Vault / Secrets Manager | Конфіг-шар |
| **Rate limiting per-org/user** | Один запит може з'їсти ресурси/бюджет усіх | API-middleware |

### 🟦 Блокує комплаєнс / великий регульований акаунт

| Gap | Наслідок для продажу | Спирається на seam |
|-----|----------------------|--------------------|
| **SSO/OIDC + SAML** | Без федеративної ідентичності у багато enterprise не зайти взагалі | Auth-шар (стандартні бібліотеки) |
| **Приватні LLM (on-prem інференс)** | Фінтех/оборона: «жоден байт коду не йде до зовнішнього API» | Порт `Generator`/`Embedder` уже абстраговано — потрібні дефолти під Llama/Ollama/Azure |
| **Data residency / multi-region** | GDPR та локальні регуляції на місце зберігання | Інфраструктурний рівень |
| **SLA / uptime-звітність** | Контракт вимагає гарантій доступності | Моніторинг + статус-сторінка |
| **Консистентність Postgres ↔ Qdrant** | Відомий gap: збій ingest між сторами → ручна репарація. Для SLA неприйнятно | Outbox/reconciliation поверх ідемпотентного ingest |

## Відомий технічний борг (з аудиту коду)

Це не «фічі», а речі, які треба прибрати перед серйозним продажем:

- **CORS `allow_origins=["*"]`** — прийнятно для dev, неприйнятно для будь-якого периметра клієнта.
- **Ключі LLM у `.env` / docker-compose** — потрібен secrets management до production.
- **Ручна репарація при збої ingest** між Postgres і Qdrant — немає outbox/reconciliation; ризик дрейфу даних.
- **Wide-open API** — будь-який клієнт може лістити/видаляти репо; до auth це блокер для shared-середовища.
- **Dev-режим compose** (reload, без health-tuning) — не production-grade.

## Стратегічний висновок

Послідовність не випадкова — вона повторює горизонти [03](03-product-roadmap.md):

1. **NOW** закриває 🟥 → відмикає перший платний pilot.
2. **NEXT** закриває 🟧 → відмикає річні enterprise-контракти.
3. **LATER** закриває 🟦 → відмикає великі регульовані акаунти.

**Найдорожчий і найважливіший gap — auth + org-модель**, бо від нього залежить уся монетизація через seat'и ([05](05-monetization.md)). Але він лягає на існуючий `RepoContext` як рівень вище, тож це розширення, а не переписування. Це і є наша архітектурна перевага: ми спроєктували шви заздалегідь.
